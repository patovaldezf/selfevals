selfevals · arquitectura de escala

Por qué estos bloqueadores importan antes de llevar selfevals a millones de registros

La arquitectura actual es buena para producto inicial: simple, honesta, modular y barata de operar. El problema no es que SQLite + JSON sea "malo"; el problema es que los caminos calientes de un sistema de evals son búsquedas por experimento, run, thread, case, iteración y tiempo. Hoy esos campos viven dentro de JSON opaco o se reconstruyen en memoria.

Fuentes locales: src/selfevals/storage/*, src/selfevals/api/*, docs/deploy.md · Fuentes externas: Langfuse, LangSmith, ClickHouse/OpenTelemetry docs.

00 · Resumen ejecutivo

Modelo actualSQLite

Un archivo local con WAL.

Entidad centralJSON

Escala sana hoy<100

El código documenta Fase A como baja escala.

Riesgo a millonesAlto

Scans, memoria y single-process.

Conclusión directa

No escalaría selfevals a millones de traces/iterations con la arquitectura actual. Sí la mantendría para MVP, demos, uso local, equipos pequeños y validación de producto. Antes de millones necesita una fase clara de scale architecture: Postgres para datos operacionales, tablas/indexes reales para entidades calientes, paginación SQL real, workers persistentes y broker externo para eventos live.

La buena noticia: el repo ya dejó puertas correctas. Existe StorageInterface, WorkspaceScope, object store para payloads grandes, WAL, concurrencia acotada y documentación que reconoce el límite. Esto reduce mucho el costo de migrar.

01 · Arquitectura actual: qué resuelve bien y dónde se queda corta

┌──────────────────────────────────────────────────────────────┐ │ FastAPI API │ │ ├─ queries.py reconstruye vistas para web │ │ ├─ run_launcher.py lanza threads daemon en el proceso │ │ └─ broker.py fan-out SSE en memoria │ │ │ │ SQLiteStorage │ │ ├─ entities(entity_type, id, workspace_id, timestamps, JSON)│ │ ├─ objects(pointer, workspace_id, key, hash, size) │ │ └─ WAL + optimistic concurrency │ │ │ │ Filesystem Object Store │ │ └─ payloads grandes, prompts/responses, blobs │ └──────────────────────────────────────────────────────────────┘

Qué sí está bien

  • Agilidad de esquema: añadir entidades Pydantic no requiere DDL.
  • Aislamiento por workspace: WorkspaceScope obliga a operar dentro de un workspace.
  • Objeto store: payloads grandes se pueden sacar del JSON inline.
  • Operación simple: un binario/servicio + un archivo SQLite + volumen persistente.
  • Documentación honesta: deploy advierte que no se debe escalar horizontalmente aún.

Dónde se rompe

  • Índices insuficientes: solo cubren workspace, tipo y timestamps.
  • Campos de negocio dentro de JSON: experiment_id, run_id, thread_id, iteration.
  • Consultas reconstruidas en Python: la API carga colecciones y pagina después.
  • Jobs no durables: si muere el proceso, muere el thread.
  • Live events no distribuibles: broker en memoria por proceso.

02 · Bloqueadores principales

1

2
Consultas por json_extract sin índicessqlite.py:175 convierte filtros desconocidos en json_extract(payload, '$...'). En volumen alto, eso fuerza scans por tipo/workspace.
3

04 · Cómo funcionan servicios similares hoy

El patrón moderno en plataformas de evals/observabilidad LLM no es "todo en una tabla JSON". Separan datos por perfil de acceso: operacional, analítico, eventos live, cola y blobs.

Langfuse

Su self-host describe dos contenedores de app: web y worker. Usa Postgres para workloads transaccionales, ClickHouse para traces/observations/scores, Redis/Valkey para queue/cache y S3/blob store para eventos, multimodal y exports.

PostgresClickHouseRedisS3Worker

Además documenta que Docker Compose es low-scale y que producción/alta disponibilidad debe ir por Kubernetes o infraestructura cloud.

LangSmith

Su self-host incluye frontend, backend API, platform backend, queue, playground y ACE backend. Para storage: ClickHouse en traces/feedback, PostgreSQL en operacional, Redis/Valkey para queue/cache y blob storage recomendado para producción.

Backend APIQueueClickHousePostgresBlob

El componente queue procesa traces/feedback de forma asíncrona y maneja retries ante errores de base de datos o conectividad temporal.

Observabilidad general

OpenTelemetry/observability stacks tratan spans como datos de ingest alto y consulta analítica. En producción se usan collectors, colas/buffers, sampling y datastores analíticos. ClickHouse es común porque comprime y filtra columnas de forma eficiente.

OTelCollectorsSamplingOLAPRetention

La idea clave: traces son evento/serie temporal, no solo entidad CRUD.

Regla de oro

Mantener JSON para flexibilidad está bien si los campos críticos también están promovidos a columnas. El patrón sano es columnas para acceso frecuente + payload JSON para detalle largo o compatibilidad. No hace falta matar la agilidad; hace falta indexar lo que el producto consulta todos los días.

06 · Plan de migración pragmático

1
Medir primeroCrear dataset sintético de 1M traces/iterations y benchmarks de endpoints actuales. Objetivo: latencia p50/p95, memoria y queries por request.
2
Promover campos calientes en SQLiteAntes de saltar a Postgres, se puede hacer una migración intermedia: columnas generadas o columnas normales para experiment_id, run_id, thread_id, iteration. Esto prueba el modelo de índices con menor blast radius.
3
Reescribir queries calientesHacer que list_experiments, experiment_cases, experiment_results, load_trace y load_thread usen SQL paginado/indexado.
4
Introducir PostgresStorageImplementar el contrato actual, pero con tablas explícitas para entidades calientes. Mantener payload para compatibilidad.
5
Separar workersMover runs de threads daemon a cola persistente. Empezar con RQ/Dramatiq/Celery si basta; Temporal si los workflows necesitan reanudación, señales y pasos largos.
6
Externalizar broker liveRedis streams/pubsub, NATS o Kafka según necesidad de replay. SSE debe poder recibir snapshot desde DB y live del bus.
7
OLAP para traces si el volumen mandaSi traces/spans son la mayoría de datos, usar ClickHouse para analítica y mantener Postgres como control plane.

Qué no haría todavía

  • No partiría todo en microservicios antes de tener benchmarks.
  • No eliminaría el JSON payload; lo mantendría como compatibilidad y fuente de detalle.
  • No metería Kafka si Redis Streams o NATS resuelven el volumen real.
  • No migraría traces a ClickHouse antes de saber si Postgres con índices cubre el primer orden de magnitud.

07 · Decisión recomendada

Mantener arquitectura actual si...

  • El producto sigue en fase local/MVP.
  • Hay pocos workspaces y decenas o cientos de experimentos.
  • Los runs son recuperables manualmente.
  • Una sola máquina con volumen persistente es suficiente.
  • El costo de infraestructura debe ser casi cero.

Migrar antes de millones si...

  • La UI debe abrir traces y threads en tiempo constante.
  • Hay ingest concurrente de muchos runs.
  • El servicio debe escalar horizontalmente.
  • Perder un run por restart es inaceptable.
  • Se necesitan dashboards analíticos sobre spans/scores.

Frase corta para roadmap

selfevals puede seguir con SQLite + JSON para producto inicial, pero la versión "millones" debe promover entidades calientes a tablas/indexes reales, mover runs a workers durables y sacar live events del proceso.

Esto no contradice la arquitectura actual; es su siguiente fase natural. El diseño actual ya insinúa esa ruta: StorageInterface, object store, comentarios de "cuando crezca", y deploy docs que dicen explícitamente dónde está la frontera.
